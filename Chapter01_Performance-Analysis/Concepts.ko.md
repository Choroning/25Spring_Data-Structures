## 1. 성능 분석 개요

### 1.1 성능이란?

프로그램의 성능은 두 가지 핵심 자원으로 측정된다:

- 프로그램이 사용하는 **공간**(메모리)
- 실행 중 소비되는 **시간**(CPU 사이클 / 경과 시간)

성능 분석은 프로그램을 실행하기 **이전에** 그 동작을 예측하는 데 도움을 주며, 성능 측정은 실제 실행으로부터 **경험적** 데이터를 제공한다.

### 1.2 왜 성능을 분석하는가?

- 동일한 문제에 대한 대안 알고리즘을 비교
- 입력 크기 증가에 따른 확장성 예측
- 배포 전 병목 지점 식별
- 정보에 기반한 설계 결정 수행

> **핵심 포인트:** 성능 분석은 *사전적(a priori)* 추정이며, 성능 측정은 *사후적(a posteriori)* 검증이다.

## 2. 공간 복잡도

### 2.1 공간의 구성 요소

프로그램 $P$에 필요한 총 공간 $S(P)$는 다음과 같다:

$$S(P) = c + S_p(\text{instance characteristics})$$

여기서:
- $c$ = **고정 부분**: 코드 공간, 단순 변수, 상수
- $S_p$ = **가변 부분**: 특정 문제 인스턴스에 따라 달라짐(입력 크기, 재귀 깊이, 동적 할당)

### 2.2 고정 공간 vs. 가변 공간

| 구성 요소 | 고정 | 가변 |
|-----------|------|------|
| 명령어 공간 | 예 | 아니오 |
| 단순 변수 | 예 | 아니오 |
| 동적 할당 메모리 | 아니오 | 예 |
| 재귀 스택 공간 | 아니오 | 예 |

### 2.3 예제: 배열 합계

```cpp
// 공간 복잡도 분석
int sum(int a[], int n) {
    int total = 0;          // 1 int
    for (int i = 0; i < n; i++)  // 1 int (i)
        total += a[i];
    return total;
}
// S_p = 0 (n에 비례하는 추가 공간 없음)
// 총합: O(1) 추가 공간 (입력 배열 제외)
```

### 2.4 재귀적 공간

```cpp
int recursiveSum(int a[], int n) {
    if (n == 0) return 0;
    return a[n-1] + recursiveSum(a, n-1);
}
// 각 재귀 호출이 스택 공간을 사용
// 재귀 깊이 = n
// S_p = O(n)
```

> **핵심 포인트:** 재귀 알고리즘은 호출 스택으로 인해 공간 복잡도가 상당히 높아질 수 있다.

## 3. 시간 복잡도

### 3.1 연산 횟수

시간 복잡도는 입력 크기 $n$의 함수로서 **핵심 연산**의 횟수를 계수하여 추정한다.

핵심 연산의 예:
- 비교
- 산술 연산
- 데이터 이동(대입)
- 함수 호출

### 3.2 단계 계수법

**프로그램 단계(program step)** 란 인스턴스 특성에 독립적인 실행 시간을 가지는 프로그램의 구문적 또는 의미적으로 유의미한 단위이다.

```cpp
// 단계 계수 예제: 행렬 덧셈
void matrixAdd(int A[][MAX], int B[][MAX], int C[][MAX], int n) {
    for (int i = 0; i < n; i++)         // n+1 회
        for (int j = 0; j < n; j++)     // n(n+1) 회
            C[i][j] = A[i][j] + B[i][j]; // n^2 회
}
// 총 단계 수: n+1 + n(n+1) + n^2 = 2n^2 + 2n + 1
// 시간 복잡도: O(n^2)
```

### 3.3 최선, 최악, 평균 경우

주어진 입력 크기 $n$에 대해:
- **최선 경우(Best case)**: 크기 $n$의 모든 가능한 입력에 대한 최소 단계 수
- **최악 경우(Worst case)**: 최대 단계 수 (가장 일반적으로 사용됨)
- **평균 경우(Average case)**: 기대 단계 수 (확률 분포 필요)

```cpp
// 순차 검색 예제
int sequentialSearch(int a[], int n, int x) {
    for (int i = 0; i < n; i++)
        if (a[i] == x) return i;
    return -1;
}
// 최선 경우:  O(1)    — 첫 번째 위치에서 발견
// 최악 경우:  O(n)    — 찾지 못하거나 마지막 위치
// 평균 경우:  O(n/2) = O(n)
```

## 4. 성능 측정

### 4.1 경과 시간 측정

```cpp
#include <chrono>

auto start = std::chrono::high_resolution_clock::now();
// ... 측정할 코드 ...
auto end = std::chrono::high_resolution_clock::now();

auto duration = std::chrono::duration_cast<std::chrono::microseconds>(end - start);
std::cout << "Time: " << duration.count() << " microseconds" << std::endl;
```

### 4.2 CPU 시간 vs. 경과 시간

| 지표 | 설명 | 사용 사례 |
|------|------|-----------|
| 경과 시간(Wall-clock time) | 실제 경과된 총 시간 | 최종 사용자 경험 |
| CPU 시간 | 해당 프로세스에 CPU가 소비한 시간 | 알고리즘 비교 |
| 사용자 시간(User time) | 사용자 모드에서의 CPU 시간 | 순수 계산 |
| 시스템 시간(System time) | 커널 모드에서의 CPU 시간 | I/O 연산 |

### 4.3 측정 모범 사례

1. 분산을 줄이기 위해 **여러 번 반복** 실행
2. 타이밍 전에 캐시를 **워밍업**
3. 일관되게 **최적화를 비활성화**(또는 활성화 상태에서 측정)
4. 유의미한 측정을 위해 **충분히 큰 입력** 사용
5. **시스템 부하**와 백그라운드 프로세스를 고려

## 5. 실용적인 프로파일링 기법

### 5.1 `clock()`을 이용한 CPU 시간 측정

```cpp
#include <ctime>

clock_t start = clock();
// ... 측정할 코드 ...
clock_t end = clock();
double cpu_time = (double)(end - start) / CLOCKS_PER_SEC;
```

### 5.2 경험적 복잡도 결정

서로 다른 입력 크기에서 실행 시간을 측정함으로써 성장률을 경험적으로 결정할 수 있다:

| n | T(n) | T(n)/n | T(n)/n^2 | T(n)/n*log(n) |
|---|------|--------|----------|---------------|
| 1000 | 0.01s | 1e-5 | 1e-8 | 1.4e-6 |
| 2000 | 0.04s | 2e-5 | 1e-8 | 2.6e-6 |
| 4000 | 0.16s | 4e-5 | 1e-8 | 4.8e-6 |

$T(n)/n^2$가 대략 일정하게 유지되면, 해당 알고리즘은 $O(n^2)$이다.

## 부록

### 성능 분석 시 흔한 실수

1. **상수 무시**: $O$-표기법은 상수를 숨기지만, 중간 크기의 $n$에서는 실제로 중요함
2. **분할 상환과 최악 경우 혼동**: 일부 연산은 가끔 비용이 크지만 평균적으로는 저렴함
3. **캐시 효과**: 메모리 접근 패턴이 실세계 성능에 상당한 영향을 미침
4. **컴파일러 최적화**: 죽은 코드를 제거하거나 연산 순서를 변경할 수 있음

### 면접 질문

1. **Q:** 시간 복잡도와 공간 복잡도의 차이는 무엇인가?
   **A:** 시간 복잡도는 입력 크기의 함수로서 연산 횟수를 측정하고, 공간 복잡도는 필요한 메모리를 측정한다.

2. **Q:** 왜 일반적으로 최악 경우 분석에 초점을 맞추는가?
   **A:** 필요한 최대 자원에 대한 보장을 제공하며, 이는 실시간 시스템과 신뢰성에 매우 중요하다.

3. **Q:** 시간 복잡도가 더 높은 알고리즘이 실제로 더 빠를 수 있는가?
   **A:** 그렇다. 작은 입력의 경우, 상수가 작은 $O(n^2)$ 알고리즘이 상수가 큰 $O(n \log n)$ 알고리즘보다 빠를 수 있다.
