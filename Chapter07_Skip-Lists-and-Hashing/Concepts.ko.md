## 1. Dictionary ADT

### 1.1 정의

**딕셔너리(dictionary)**는 각 원소가 **키(key)**라는 필드를 가진 원소들의 모음이다. 모든 원소는 **(key, value)** 쌍으로 저장되며, 모든 키는 보통 고유(unique)하다.

> **핵심:** 딕셔너리는 키를 값에 매핑하는 추상 자료형으로, 효율적인 검색, 삽입, 삭제를 지원한다.

### 1.2 딕셔너리 연산

대표적인 딕셔너리 연산은 다음과 같다:

1. **empty()** — 딕셔너리가 비어 있는지 판별
2. **size()** — (key, value) 쌍의 개수 반환
3. **insert(key, value)** — 새로운 쌍을 딕셔너리에 삽입
4. **search(key)** — 지정된 키를 가진 쌍을 탐색
5. **delete(key)** — 지정된 키를 가진 쌍을 삭제

### 1.3 중복이 있는 딕셔너리

일부 응용에서는 키가 고유할 필요가 **없다**. 단어 사전이 대표적인 예이다:

- 쌍은 `(단어, 의미)` 형태
- 하나의 단어에 여러 항목이 있을 수 있다 (예: "rank"는 사회적 지위, 공식 등급, 순서 매기기 등의 의미를 가질 수 있음)

### 1.4 딕셔너리 원소 접근

두 가지 접근 패턴이 있다:

| 접근 유형 | 설명 |
|:----------|:-----|
| **임의 접근 (Random Access)** | 키에 대한 탐색을 수행하여 임의의 원소를 검색 |
| **순차 접근 (Sequential Access)** | `Begin()`과 `Next()`를 사용하여 키 필드의 오름차순으로 원소를 하나씩 검색 |

### 1.5 정렬된 선형 리스트로서의 딕셔너리

$L = (e_1, e_2, e_3, \ldots, e_n)$이 주어지고 각 $e_i$가 `(key, value)` 쌍일 때:

| 표현 방법 | 탐색 시간 |
|:----------|:----------|
| 비정렬 배열 | $O(n)$ |
| 정렬 배열 | $O(\log n)$ — 이진 탐색 |
| 비정렬 체인 | $O(n)$ |
| 정렬 체인 | $O(n)$ |

> **핵심:** 정렬 배열은 이진 탐색을 통해 $O(\log n)$ 탐색이 가능하지만, 정렬 체인은 중간에 인덱싱할 수 없으므로 여전히 $O(n)$이다. 스킵 리스트가 이 문제를 해결한다.

## 2. 스킵 리스트

### 2.1 동기

- **정렬 배열**에서의 이진 탐색은 $O(\log n)$이지만, 삽입/삭제는 이동 때문에 $O(n)$
- **정렬 체인**은 위치를 찾으면 $O(1)$ 삽입/삭제가 가능하지만, 탐색이 $O(n)$

**질문:** 정렬 체인의 탐색 성능을 어떻게 개선할 수 있는가?

**답:** 일부 노드에 **추가적인 전방 포인터**를 부여한다 — 이렇게 확장된 체인을 **스킵 리스트**라 한다.

### 2.2 구조와 레벨

스킵 리스트는 여러 레벨의 정렬 체인으로 구성된다:

- **레벨 0** 체인은 **모든 원소**를 포함
- **레벨 1** 체인은 매 **2번째** 원소를 포함
- **레벨 2** 체인은 매 **4번째** 원소를 포함
- **레벨 $i$** 체인은 매 $2^i$번째 원소를 포함

어떤 원소가 레벨 0부터 레벨 $i$까지의 체인에 모두 나타나는 경우에만 해당 원소를 **레벨 $i$ 원소**라 한다.

```
Level 2:  H ──────────────────> 40 ──────────────────> 80 ──> T
Level 1:  H ──────> 24 ──────> 40 ──────> 60 ──────> 80 ──> T
Level 0:  H ──> 20 ──> 24 ──> 30 ──> 40 ──> 60 ──> 75 ──> 80 ──> T
```

> **핵심:** $n$개 원소에 대한 레벨 수는 $\lfloor \log_{1/p} n \rfloor + 1 = \lfloor \log_p n \rfloor + 1$이다. $p = 1/2$이고 8개 원소일 때: $\log_2 8 = 3$ 레벨.

### 2.3 탐색 연산

키 $k$를 탐색하려면 헤드 노드의 **최상위 레벨**에서 시작하여 오른쪽으로 순회한다:

1. 현재 레벨에서 다음 노드의 키가 $k$보다 작은 동안 오른쪽으로 이동
2. 다음 노드의 키가 $k$와 같으면 원소를 찾음
3. 다음 노드의 키가 $k$를 초과하면(또는 꼬리이면) 한 레벨 아래로 이동
4. 찾거나 레벨 0이 끝날 때까지 반복

**예시:** 3레벨 스킵 리스트에서 77을 탐색하면 최대 $\log_2 n = 3$번의 비교가 필요하다:

1. 레벨 2에서 시작: 40과 비교(오른쪽 이동), 80과 비교(아래로 이동)
2. 레벨 1에서: 60과 비교(오른쪽 이동), 80과 비교(아래로 이동)
3. 레벨 0에서: 75와 비교(오른쪽 이동), 80과 비교 → 77 발견되지 않음 또는 여기에 삽입

### 2.4 삽입과 삭제

**삽입:**
- 새 원소가 삽입될 때, 확률 $p^i$ (일반적으로 $p = 1/2$)에 따라 레벨이 지정
- 새 원소가 레벨 $i$ 원소일 확률은 $1/2^i$
- 레벨을 결정한 후, 영향을 받는 각 레벨에서 전방 포인터를 갱신

**삭제:**
- 탐색 절차를 사용하여 원소를 찾음
- 해당 원소가 나타나는 모든 레벨에서 제거
- 각 레벨에서 이전 노드의 전방 포인터를 갱신

> **핵심:** 삽입과 삭제는 스킵 리스트 구조를 유지하기 위해 최악의 경우 $O(n)$이 필요하지만, 기대 시간은 $O(\log n)$이다.

### 2.5 최악의 경우 시간 복잡도

모든 원소가 **레벨 0에만** 있는 경우(상위 레벨 포인터 없음), 스킵 리스트는 단순 정렬 연결 리스트로 퇴화한다. 이 경우:

$$T_{\text{worst}}(n) = \Theta(n)$$

### 2.6 평균적인 경우 시간 복잡도

**기대** 탐색 시간을 분석하기 위해 탐색 경로를 **역방향**(위쪽과 왼쪽으로)으로 추적한다:

$C(i)$를 $i$개 레벨을 역방향으로 올라가는 기대 비용이라 하자. 각 노드에서:

- **경우 1** (확률 $1 - p$): 노드에 상위 레벨 포인터가 없음 → 비용은 $C(i) + 1$
- **경우 2** (확률 $p$): 노드에 상위 레벨 포인터가 있음 → 비용은 $C(i-1) + 1$

$$C(i) = (1-p)(1 + C(i)) + p(1 + C(i-1))$$

$C(0) = 0$으로 점화식을 풀면:

$$C(i) = \frac{i}{p}$$

최대 레벨 $i = \log_p n = \log n$이므로:

$$\Theta(C(i)) = \Theta\left(\frac{\log N}{p}\right) = \Theta(\log N) = O(\log N)$$

> **시험 팁:** 평균적인 경우의 분석은 고전적인 점화식 문제이다. 기억할 것: $p = 1/2$일 때, 평균 탐색 시간은 $O(\log n)$ — 본질적으로 정렬 배열에서의 이진 탐색과 동일하지만, 효율적인 삽입/삭제가 가능하다.

## 3. 해싱

### 3.1 이상적 해싱

**이상적 해싱**(또는 직접 주소 지정)은 키 자체를 배열 인덱스로 사용한다:

- 키 $k$를 가진 원소를 `table[k]`에 저장
- 탐색, 삽입, 삭제 모두 $O(1)$

**제한:** 키 범위가 매우 클 경우(예: 9자리 학번), 테이블이 $10^9$개의 슬롯을 필요로 하여 비현실적이다.

### 3.2 해시 함수

**해시 함수** $h(k)$는 키를 더 작은 테이블 인덱스 범위 $[0, m-1]$로 매핑한다:

$$h: K \rightarrow \{0, 1, 2, \ldots, m-1\}$$

**대표적인 해시 함수:**

| 함수 | 공식 | 비고 |
|:-----|:-----|:-----|
| 나눗셈(Division) | $h(k) = k \mod m$ | $m$은 2의 거듭제곱에 가깝지 않은 소수로 선택 |
| 곱셈(Multiplication) | $h(k) = \lfloor m(kA \mod 1) \rfloor$ | $A \approx 0.6180$ (황금비) |
| 접기(Folding) | 키를 부분으로 나누어 합산 | 문자열 키에 적합 |

> **핵심:** **충돌(collision)**은 서로 다른 두 키가 같은 인덱스에 매핑될 때 발생한다: $h(k_1) = h(k_2)$이고 $k_1 \neq k_2$. 충돌 해결이 해싱의 핵심 과제이다.

### 3.3 충돌 해결 — 개방 주소법

개방 주소법에서는 모든 원소가 해시 테이블 자체 내에 저장된다. 충돌이 발생하면 다음 빈 슬롯을 탐사한다:

**선형 탐사(Linear Probing):**
$$h_i(k) = (h(k) + i) \mod m, \quad i = 0, 1, 2, \ldots$$

- 단순하지만 **1차 클러스터링(primary clustering)** 발생 — 연속 점유 셀의 긴 체인이 형성

**이차 탐사(Quadratic Probing):**
$$h_i(k) = (h(k) + i^2) \mod m, \quad i = 0, 1, 2, \ldots$$

- 1차 클러스터링을 줄이지만 **2차 클러스터링(secondary clustering)** 발생 가능

**이중 해싱(Double Hashing):**
$$h_i(k) = (h_1(k) + i \cdot h_2(k)) \mod m$$

- 두 번째 해시 함수를 사용하여 탐사 간격을 결정
- 개방 주소법 중 가장 좋은 분포

### 3.4 충돌 해결 — 체이닝

**분리 체이닝(separate chaining)**에서 각 테이블 슬롯은 해당 인덱스로 해싱된 모든 원소의 연결 리스트에 대한 포인터를 보유한다:

```
Index 0: → [20] → [40] → NULL
Index 1: → [31] → NULL
Index 2: → [12] → [52] → NULL
...
```

- 클러스터링 문제 없음
- 테이블이 크기보다 더 많은 원소를 보유할 수 있음 (적재율 $\alpha > 1$ 가능)
- 각 체인은 정렬 또는 비정렬 연결 리스트 가능

### 3.5 성능 비교

$\alpha = n/m$을 **적재율(load factor)**(원소 수 대 테이블 크기의 비율)이라 하자:

| 방법 | 성공적 탐색 | 실패한 탐색 |
|:-----|:-----------|:-----------|
| 선형 탐사 | $\frac{1}{2}\left(1 + \frac{1}{1-\alpha}\right)$ | $\frac{1}{2}\left(1 + \frac{1}{(1-\alpha)^2}\right)$ |
| 이차 / 이중 해싱 | $-\frac{1}{\alpha}\ln(1-\alpha)$ | $\frac{1}{1-\alpha}$ |
| 체이닝 | $1 + \frac{\alpha}{2}$ | $1 + \alpha$ |

> **핵심:** 적재율이 높을 때 체이닝이 일반적으로 개방 주소법보다 우수하다. 낮은 적재율($\alpha < 0.5$)에서는 모든 방법의 성능이 유사하다.

## 요약

| 개념 | 핵심 |
|:-----|:-----|
| Dictionary ADT | (key, value) 쌍의 모음으로 탐색, 삽입, 삭제 지원 |
| 스킵 리스트 | 다중 레벨 연결 리스트로 기대 탐색/삽입/삭제 $O(\log n)$ |
| 스킵 리스트 레벨 | 레벨 $i$는 매 $2^i$번째 원소 포함; 총 레벨 = $\lfloor \log_2 n \rfloor + 1$ |
| 스킵 리스트 최악의 경우 | $\Theta(n)$ — 연결 리스트로 퇴화 |
| 스킵 리스트 평균적 경우 | $O(\log n)$ — 역방향 분석 점화식으로 유도 |
| 해시 함수 | 키를 테이블 인덱스로 매핑; 충돌 해결 필요 |
| 개방 주소법 | 테이블 내에서 선형, 이차, 또는 이중 해싱 탐사 |
| 체이닝 | 각 슬롯이 충돌 원소의 연결 리스트를 보유 |
| 적재율 $\alpha$ | $n/m$; 값이 높을수록 충돌 빈도 증가 |

## 부록

### 실세계 응용

- **해시 맵** (C++ `unordered_map`, Java `HashMap`, Python `dict`)은 내부적으로 체이닝 또는 개방 주소법을 사용
- **스킵 리스트**는 Redis(정렬 집합), LevelDB, 동시성 자료구조에서 사용
- **블룸 필터(Bloom Filter)**는 확률적 멤버십 테스트를 위해 해싱을 확장

### 면접 팁

- 개방 주소법과 체이닝의 차이점을 숙지할 것
- 충돌 확률과 기대 탐사 길이를 분석할 수 있어야 함
- 스킵 리스트는 시스템 설계에서 균형 BST의 일반적인 대안
